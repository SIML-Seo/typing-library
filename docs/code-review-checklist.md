# 코드 리뷰 체크리스트

버전: v0.1 (Draft)
작성일: 2026-03-06

## 0) 목적

이 문서는 PR 리뷰에서 **과한 방어코딩**, **silent failure**, **도메인 로직 비대화**를 빠르게 걸러내기 위한 체크리스트다.

## 1) 리뷰 우선순위

리뷰는 아래 순서로 본다.

1. 경계 검증이 적절한가
2. 내부 로직이 불필요하게 복잡해지지 않았는가
3. fallback이 버그를 숨기지 않는가
4. 에러 상태가 사용자/개발자 관점에서 분리되어 있는가
5. 제품 정책이 코드에 명확히 드러나는가

## 2) 경계 검증 체크

- 외부 입력(폼, 업로드, URL, fetch 응답, IndexedDB)이 검증되는가
- 검증이 boundary에서 한 번만 일어나는가
- 형식 오류와 빈 데이터가 구분되는가
- 시스템 실패와 사용자 입력 오류가 같은 메시지로 합쳐지지 않았는가

## 3) 과한 방어코딩 체크

- 내부 함수마다 같은 `null` 체크가 반복되지 않는가
- 이미 타입이 보장된 값에 `typeof`, `!value`, `Array.isArray`를 계속 쓰지 않는가
- `try/catch` 범위가 너무 넓지 않은가
- `catch`에서 조용히 기본값을 반환하지 않는가
- `?.`, `??`, `||`가 계약 위반을 숨기지 않는가

## 4) fallback 체크

- 이 fallback이 실제 제품 정책인가
- preview 모드 같은 명시적 정책이 아닌데 기본값으로 뭉개고 있지 않은가
- 에러인데도 성공처럼 보이게 만들지 않았는가
- UI fallback과 데이터 fallback이 섞여 있지 않은가

## 5) 모듈별 체크 포인트

### 5.1 `src/shared/lib/*`

- fetch 실패를 `[]`, `{}`로 덮지 않았는가
- 파싱 오류를 명시적으로 드러내는가
- 환경변수 누락을 조용히 숨기지 않는가

### 5.2 `src/shared/db/*`

- IndexedDB 지원 불가 / open 실패 / transaction 실패를 구분하는가
- 저장소 레이어가 UI fallback까지 책임지고 있지 않은가
- 저장 실패를 성공처럼 처리하지 않는가

### 5.3 `src/features/library/*`

- catalog 오류가 “작품 0개”처럼 보이지 않는가
- preview fallback이 정책 범위 안에 있는가
- works origin 미설정과 works origin 장애가 구분되는가

### 5.4 `src/features/typing/*`

- 입력 판정 함수가 boundary 검증까지 떠안고 있지 않은가
- 계산 함수가 불필요한 fallback으로 오염되지 않았는가
- 문단 엔진이 UI 상태 복구 책임까지 같이 지지 않는가

### 5.5 `src/features/results/*`

- 결과 저장 실패가 조용히 무시되지 않는가
- 저장 성공/실패 상태가 사용자에게 분명한가

## 6) 리뷰에서 바로 잡아야 하는 패턴

- `catch { return [] }`
- `catch { return null }`
- `catch { return '' }`
- `return defaultValue`가 너무 많아 실제 실패가 보이지 않는 코드
- 내부 순수 함수에 과도한 입력 검증이 반복되는 코드
- 정책 fallback과 버그 은폐 fallback이 구분되지 않는 코드

## 7) 리뷰 코멘트 템플릿

### 7.1 과한 guard

- “이 값은 상위 boundary에서 이미 검증된 상태라면, 여기 guard는 중복입니다. 내부 로직을 단순하게 유지하는 편이 낫습니다.”

### 7.2 silent fallback

- “여기 fallback이 네트워크/파싱 오류를 정상 상태처럼 숨기고 있습니다. 사용자 fallback UI와 데이터 실패 처리를 분리하는 쪽이 안전합니다.”

### 7.3 넓은 try/catch

- “`try/catch` 범위가 너무 넓어 실패 원인이 사라집니다. fetch/parse/store 단계별로 실패를 분리하는 편이 좋습니다.”

### 7.4 정책 불명확

- “이 기본값이 제품 정책인지, 단순 편의인지 코드만으로는 불분명합니다. 정책이면 이름/주석/함수 경계로 드러내는 편이 좋습니다.”

## 8) 승인 전 마지막 질문

- 이 코드가 장애를 줄이는가, 아니면 버그를 숨기는가?
- boundary 검증과 내부 로직이 분리되어 있는가?
- fallback 이유를 한 문장으로 설명할 수 있는가?
- 실패했을 때 운영자가 원인을 추적할 수 있는가?
- 이 PR이 `typing` 엔진을 불필요하게 무겁게 만들지 않았는가?

## 9) 현재 프로젝트 기준 리뷰 원칙

- 경계는 엄격하게 본다
- 내부는 단순함을 우선한다
- preview 모드는 허용하지만 silent failure는 허용하지 않는다
- “안전해 보이는 복잡함”보다 **명확한 계약과 명시적 실패**를 더 높게 평가한다
