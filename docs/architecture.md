# 아키텍처 / 모듈 경계

버전: v0.1 (Draft)
작성일: 2026-03-05

## 0) 목표

- “필사형 타이핑” 핵심 기능을 빠르게 구현하되, **무조건 무료(운영비 0원 보장)** 제약을 만족한다.
- MVP는 **정적 사이트(서버 런타임/DB 없음)** 로 배포한다.
- 공개 작품(정적)과 사용자 데이터(로컬)를 명확히 분리한다.
- 저장소/배포 경계를 문서로 고정해, 기능 추가 시 비용/보안 리스크를 줄인다.

## 1) 시스템 구성(MVP: 정적)

- Frontend: 웹 앱(정적 배포)
  - 현재 레포는 Next.js(App Router) 기반이지만, MVP는 “서버 런타임 없이” 동작하도록 구성한다(정적 export 또는 SPA 전환).
- SEO: 랜딩(마케팅) 페이지 중심
  - 검색 노출은 랜딩까지만(앱 기능 화면은 비대상, 예: noindex)
- Hosting: Firebase Hosting (1순위)
  - 원칙: **Spark 플랜 + Hosting만** 사용(Functions/Firestore/Storage 등 서버 기능 금지)
  - 가드레일: 결제수단(카드) 연결이 필요하거나, 초과 사용 시 과금이 가능한 정책이 확인되면 즉시 제외/대체한다(0원 보장 우선).
  - 백업 후보: Cloudflare Pages / GitHub Pages / Vercel(정적 배포만)
    - Vercel 주의: Next.js의 서버 기능(Functions/Edge/SSR 등)과 결합되기 쉬우므로, MVP에서는 “정적 export(서버 런타임 0)”로만 제한한다.
- Public Works(공개 작품)
  - **작품 전용 정적 호스팅(works origin)** 으로 분리한다(앱 배포와 독립적으로 업데이트).
    - 1순위: Firebase Hosting 내 별도 사이트(예: `works` 사이트)
    - 앱은 `worksBaseUrl`(예: `https://<project>-works.web.app`)에서 인덱스/원문을 fetch 한다
    - (필요 시) 다른 origin에서 fetch 하므로 works 호스팅에 CORS 헤더를 설정한다(공개 데이터이므로 `*` 허용 가능)
    - 저장소 운영: 작품 원문은 별도 git 저장소(works repo)로 관리하고, 앱 레포에는 `works/` **git submodule** 로 연결한다(모노레포처럼 작업하되 히스토리는 분리).
  - 작품 메타데이터 인덱스(정적 JSON)
    - 예: `GET {worksBaseUrl}/works/index.json`
  - 작품 원문(정적 텍스트 파일)
    - 예: `GET {worksBaseUrl}/works/{workId}.txt`
    - 대용량 작품은 “파트(장/권/분할 파일)”로 쪼개 제공할 수 있어야 한다(인덱스에 parts 메타데이터 포함)
- Local(사용자 데이터)
  - IndexedDB: 내 작품 + 결과(히스토리) + 진행 중 세션 드래프트 저장(기본)
  - LocalStorage: 설정/요약 저장(보조)
- Analytics: GA4(DAU 측정, 콘텐츠/PII 전송 금지)
- Ads(옵션): Google AdSense 스크립트 삽입(정적 사이트에서도 가능)

## 2) 데이터 흐름(MVP)

### 2.1 공개 작품 조회(모든 사용자)

1) 앱이 작품 인덱스(예: `{worksBaseUrl}/works/index.json`)를 로드한다.
2) 사용자가 작품을 선택하면 원문 파일(예: `{worksBaseUrl}/works/{workId}.txt`)을 fetch 한다.
3) 필사 화면에서 **문단(=페이지/Chunk) 단위**로 분절하여 표시한다.

### 2.2 진행 중 세션 자동 저장(모든 사용자)

- 필사 중인 문단/입력값/누적 시간/설정 스냅샷을 로컬 IndexedDB에 저장한다.
- 작품 진입 시 저장된 드래프트가 있으면 “이어하기”를 제안한다.

### 2.3 결과 저장(모든 사용자)

- 결과는 로컬 IndexedDB에 저장한다(서버 전송 없음).
- “내 기록” 화면은 로컬 DB에서 조회한다.

### 2.4 내 작품 업로드(모든 사용자)

- 사용자가 업로드한 텍스트는 로컬 IndexedDB에 저장한다.
- 내 작품은 공개 라이브러리와 동일한 필사 엔진을 사용한다.

### 2.5 공개 작품 추가(운영 프로세스)

1) 콘텐츠 담당자가 작품 텍스트를 준비한다(저작권 만료 근거 포함).
2) works repo(서브모듈)의 정적 디렉터리에 작품 원문/메타데이터를 추가한다(예: `works/public/works/*`).
3) (권장) 인덱스 JSON을 자동 생성/검증(스크립트)하고, works 호스팅만 배포한다(예: `firebase deploy --only hosting:works`).

## 3) 권한/인증 경계

- MVP는 서버 인증/권한 시스템을 두지 않는다(정적 사이트).
  - “게스트 필사/내 작품/결과”는 로컬 저장이므로 인증이 필수는 아니다.
- 추후(선택) 로그인/권한이 필요해지면, “0원 보장(자동 과금 0)”과 충돌하지 않는 방식으로 별도 ADR로 결정한다.

## 4) 모듈 경계(권장)

> 구현이 시작되면 아래처럼 “도메인 단위”로 경계를 두는 것을 권장한다.

- `works` (작품)
  - 공개 작품 리포지토리(정적 인덱스 + 텍스트 fetch)
  - 내 작품 리포지토리(IndexedDB)
- `typing` (필사 엔진)
  - 텍스트 분절(페이지)
  - 입력 처리(IME 포함)
  - 오타 정책(진행 허용/즉시 강조/페이지 리포트)
  - 지표 계산(WPM/정확도/시간)
- `results` (결과)
  - 로컬 저장/조회(IndexedDB)
- (후속 선택) `auth` (인증)
  - 기기간 동기화/서버 저장/계정 기반 기능이 필요해질 때만 도입
  - MVP에서는 사용하지 않는다(정적 사이트)
- `analytics` (DAU)
  - GA4 이벤트 전송(콘텐츠/토큰/PII 전송 금지)

## 5) 무료 운영 관점의 의사결정 가이드

- “서버 쓰기”는 비용의 직접 원인이므로 MVP에서는 서버 저장을 없다시피 한다.
- 공개 작품 제공은 읽기 트래픽이 발생하므로, 정적 파일 분할/캐싱(브라우저 캐시/Service Worker)으로 완화한다.
- 유저 데이터(결과/내 작품)는 로컬 저장으로 시작한다.
- “0원 보장”을 위해 결제수단 연결/초과 과금이 가능한 인프라는 지양하고, 무료 한도 초과 시 차단/제한되는 하드 캡을 우선한다.

## 6) 운영 가드레일

- Firebase Hosting은 `Spark + Hosting만` 유지한다.
  - 공개 런칭 전에 결제수단/유료 플랜 요구가 확인되면 Firebase를 배포 대상에서 제외하고 백업 호스트로 전환한다.
- works repo 운영은 아래 흐름으로 고정한다.
  1. works repo에서 작품 원문/메타데이터를 수정한다
  2. 인덱스 생성/검증 스크립트를 실행한다
  3. works 호스팅만 별도로 배포한다
  4. 필요하면 앱 레포에서 submodule 포인터를 갱신해 재현성을 맞춘다
- 공개 작품 전송량은 “앱과 작품 분리 + 파트 분할 + 캐시 헤더”로 관리한다.
  - 인덱스는 짧은 캐시 또는 `no-cache`
  - 원문/파트 파일은 버전/해시 기반 파일명과 장기 캐시 사용
- 작품 메타데이터 인덱스에는 최소한 작품 ID, 제목, 저자, 언어, 저작권 만료 근거, 원문 경로를 포함한다.

## 7) 구현 전환 전략

- 새 레포를 만들지 않고, **현재 레포 안에서 MVP 구현 영역을 분리**한다.
- 현재 AWS 프로토타입은 즉시 확장하지 않고 레거시로 취급한다.
- 구현 순서는 아래를 권장한다.
  1. 새 MVP 라우트/기능 모듈을 별도 구조로 추가
  2. 새 MVP가 핵심 플로우를 대체하면 기존 진입 경로를 제거 또는 격리
  3. 마지막으로 Amplify/NextAuth 의존 코드와 관련 환경을 정리

권장 목표 구조(초안)
- `src/app`: 새 MVP 라우트 레이어
- `src/features/*`: 작품/필사/결과/설정 등 MVP 기능 모듈
- `src/shared/db/*`: IndexedDB 스키마/레코드 타입/공통 접근 레이어
- `src/shared/store/*`: `zustand` 기반 전역 UI 상태
- `src/shared/*`: 그 외 공통 UI/유틸
- `src/legacy/*`: 이전 프로토타입 코드 임시 보관 영역(필요 시)
- `amplify/`: 레거시 인프라 정의(동결 후 제거 대상)
- 구체적인 단계별 전환 순서는 `/docs/mvp-implementation-plan.md`를 따른다

원칙
- 새 MVP 구현은 AWS/NextAuth/Amplify 경로를 사용하지 않는다.
- 영속 데이터의 source of truth는 `IndexedDB`이며, `zustand`는 얇은 UI 상태에만 사용한다.
- 레거시 변경은 “격리, 제거, 마이그레이션” 목적에 한정한다.

## 8) 레거시(현재 프로토타입)

레포에는 Amplify Gen2(Auth/Data/Storage) + NextAuth/Cognito 기반 프로토타입 코드가 남아있을 수 있다.
하지만 MVP는 “정적 + 0원 보장” 원칙에 따라 **AWS를 사용하지 않으므로**, 해당 프로토타입은 정리/제거 대상이다.
