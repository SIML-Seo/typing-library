# ADR 0001: 오타 처리 정책(진행 허용 + 페이지 리포트)

상태: Accepted  
작성일: 2026-03-05  
관련 문서: `/docs/requirements.md`, `/docs/ux-flows.md`

## 컨텍스트

- 필사형 타이핑에서 오타 처리 방식은 UX의 핵심이다.
- “오타가 날 때마다 진행을 막으면” 학습 흐름이 끊길 수 있다.
- 반대로 “오타를 무시하면” 학습 피드백이 약해진다.

## 결정

- 오타가 발생해도 입력은 계속 허용한다.
- 오타가 발생한 위치는 즉시 강조한다(예: 빨간색).
- 페이지(Chunk) 종료 시, 해당 페이지의 오타를 요약해 리포트한다.

### 판정/편집 규칙(세부)

- 비교는 “위치 기반(문자 단위)”으로 한다: `typed[i]` vs `target[i]`
- 공백(space)은 원문과 **완전 일치**해야 한다.
- 줄바꿈(newline)은 원문과 **완전 일치**해야 한다.
  - 목표 텍스트에 줄바꿈이 포함된 경우, 사용자는 해당 위치에서 Enter 입력으로 줄바꿈을 만들어야 한다.
- 구두점/대문자 판정 옵션
  - 기본값: **ON**
  - ON: 구두점/대문자까지 `target[i]`와 일치해야 정타로 판정
  - OFF: 구두점/대문자 차이는 오타로 판정하지 않음(정확도/오타 계산에서 제외)
- Backspace 정책
  - Backspace로 직전 입력을 삭제/수정할 수 있다.
  - Backspace로 수정되어 더 이상 불일치가 남지 않으면, 해당 위치는 오타로 집계하지 않는다.
- 커서 정책(일반적인 타자연습 UX)
  - 입력은 항상 “현재 진행 위치”에서만 이뤄지며, 사용자가 임의로 커서를 이동해 중간을 편집하는 UX는 제공하지 않는다(화살표/마우스 이동 등은 비활성화).
- 입력 소스 정책
  - 한글 IME 조합 입력과 일반 키보드 입력은 허용한다.
  - 필사 화면에서 붙여넣기, 드래그앤드롭 텍스트 투입, 스크립트/매크로성 다문자 삽입은 허용하지 않는다.

## 대안

- 대안 A) 오타가 나면 진행을 멈추고 수정 강제(엄격)
  - 장점: 정렬/평가가 단순
  - 단점: 몰입 저하, 진행이 답답할 수 있음
- 대안 B) 오타가 나도 진행 허용 + 즉시 표시 + 페이지 리포트(채택)
  - 장점: 흐름 유지 + 피드백 유지
  - 단점: 입력 평가/표시 로직이 더 복잡해질 수 있음(특히 한글 IME/편집)

## 결과(Consequences)

- 타이핑 엔진은 “위치 기반 비교 + 오타 상태 표시 + 페이지 리포트 데이터”를 제공해야 한다.
- 페이지(Chunk) 정의가 필요하다(글자 수/문단/문장 N개 등).
- 공백/줄바꿈을 완전 일치로 판정하므로, UX 상 “공백/줄바꿈이 어디에 있는지”를 사용자가 인지할 수 있는 보조 표기가 필요할 수 있다(예: 다중 공백 표시, 줄바꿈 표시).
