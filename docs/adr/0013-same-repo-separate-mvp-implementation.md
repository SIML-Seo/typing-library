# ADR 0013: 기존 레포를 유지하고 MVP 구현 영역을 분리한다

상태: Accepted  
작성일: 2026-03-06  
관련 문서: `/docs/current-state.md`, `/docs/architecture.md`, `/docs/decisions.md`

## 컨텍스트

- 현재 레포에는 AWS(Amplify/Cognito/AppSync/S3) 기반 프로토타입이 남아 있다.
- 동시에 MVP는 **정적 사이트 + 운영비 0원 보장** 원칙으로 새로 구현해야 한다.
- 문서/운영 스크립트/works 저장소 가이드는 이미 현재 레포를 기준으로 정리되고 있다.
- 이 시점에 레포를 새로 만들면 히스토리와 문맥이 분산되고, 기존 프로토타입과 새 MVP를 비교·정리하기가 어려워진다.
- 반대로 현재 코드를 즉시 전부 덮어쓰면, 전환 중에 참고할 구현/도메인 지식까지 같이 잃을 수 있다.

## 결정

- **새 레포를 만들지 않고, 현재 레포를 그대로 유지한다.**
- MVP는 이 레포 안에서 **기존 프로토타입과 분리된 구현 영역**으로 개발한다.
- 기존 AWS 프로토타입은 즉시 확장하지 않고, 임시로 “레거시 영역”으로 취급한다.
- 구현 전환 원칙은 아래와 같다.
  1. 새 MVP 화면/로직을 우선 분리 구현한다
  2. 새 경로와 새 상태 관리가 안정화되면 기존 프로토타입 진입 경로를 제거하거나 격리한다
  3. 최종적으로 AWS 관련 레거시 코드와 인프라 의존성을 정리한다

권장 목표 구조(초안)
- `src/app`: 새 MVP 라우트 레이어
- `src/features/*`: 필사/작품/결과 등 MVP 기능 모듈
- `src/shared/*`: 공통 UI/유틸
- `src/legacy/*`: 이전 프로토타입 코드 임시 보관 영역(필요 시)
- `amplify/`: 레거시 인프라 정의(동결 후 제거 대상)

추가 원칙
- 새 MVP 구현은 AWS/NextAuth/Amplify 의존 경로를 사용하지 않는다.
- 레거시 수정은 “새 MVP 구현을 위해 꼭 필요한 격리/제거 작업”에 한정한다.

## 대안

- 대안 A) 새 레포를 만들어 MVP를 완전히 별도로 시작
  - 장점: 구조가 깔끔하고 레거시와 완전히 분리됨
  - 단점: 현재 문서/히스토리/운영 스크립트가 분산되고 전환 비용이 커짐
- 대안 B) 현재 레포를 바로 갈아엎어 덮어쓰기
  - 장점: 구조가 빨리 단순해짐
  - 단점: 전환 과정에서 비교 대상과 참고 구현을 잃기 쉬움
- 대안 C) 현재 레포 유지 + MVP 구현 영역 분리(채택)
  - 장점: 히스토리를 유지하면서 안전하게 단계적 전환 가능
  - 단점: 일정 기간 레거시와 MVP가 공존하므로 구조 관리가 필요함

## 결과(Consequences)

- 문서/스크립트/배포 준비는 현재 레포를 기준으로 계속 누적할 수 있다.
- 구현 시작 단계에서 “무엇이 레거시이고 무엇이 새 MVP인지” 폴더/모듈 경계로 명확히 표시해야 한다.
- 레거시 코드를 장기간 방치하면 혼란이 커지므로, MVP 기본 흐름이 붙는 시점에 정리 작업을 바로 이어가야 한다.
