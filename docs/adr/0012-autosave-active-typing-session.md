# ADR 0012: 진행 중 필사 세션은 자동 저장하고 이어하기를 제공

상태: Accepted  
작성일: 2026-03-06  
관련 문서: `/docs/prd.md`, `/docs/requirements.md`, `/docs/ux-flows.md`, `/docs/data-model.md`

## 컨텍스트

- 필사 앱은 긴 작품을 문단 단위로 진행하므로, 새로고침/탭 종료/브라우저 재방문으로 세션이 끊길 가능성이 높다.
- MVP는 계정/서버 동기화가 없으므로, 진행 상태 보존은 로컬 저장으로 해결해야 한다.
- 중단 시 매번 처음부터 다시 시작하게 하면 이탈률이 높아지고, DAU/완료율에도 불리하다.

## 결정

- 진행 중인 필사 세션은 로컬 IndexedDB에 자동 저장한다.
- 작품 진입 시 저장된 드래프트가 있으면 “이어하기 / 새로 시작” UI를 제공한다.
- 이어하기는 아래 상태를 복원한다.
  - 현재 작품/파트
  - 현재 문단 인덱스
  - 현재 문단 입력값
  - 누적 시간
  - 설정 스냅샷
- 작품 완료 시에는 드래프트를 삭제하고 결과(Result)만 남긴다.
- 저장된 공개 작품 버전(`workChecksum`)이 바뀐 경우, 기존 드래프트는 이어하기 대신 폐기 또는 새로 시작을 유도할 수 있다.

## 대안

- 대안 A) 자동 저장 없이 완주 결과만 저장
  - 장점: 구현이 단순
  - 단점: 중단 시 복구 불가, 긴 작품 UX가 나빠짐
- 대안 B) 서버 동기화 기반 이어하기
  - 장점: 기기 간 이어하기 가능
  - 단점: MVP의 0원 보장/무서버 원칙과 충돌
- 대안 C) 로컬 자동 저장 + 이어하기(채택)
  - 장점: 무서버 원칙 유지, 이탈 복구 가능
  - 단점: 기기 변경/브라우저 데이터 삭제 시 복구 불가

## 결과(Consequences)

- 결과 히스토리와 별도로 `진행 중 세션 드래프트` 저장소가 필요하다.
- 소요시간은 “앱이 실제로 진행 중이던 누적 시간”만 저장하고, 브라우저를 닫아둔 시간은 포함하지 않는다.
- UX에는 이어하기 진입점과 “새로 시작”에 따른 드래프트 삭제 흐름이 포함되어야 한다.
