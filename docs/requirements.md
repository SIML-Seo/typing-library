# 요구사항 / 수용기준 (MVP)

버전: v0.1 (Draft)  
작성일: 2026-03-05

## 0) 범위

이 문서는 “필사형 타이핑” MVP의 요구사항과 수용기준(acceptance criteria)을 정의한다.  
목표는 **무조건 무료(운영비 0원 보장)** 제약을 만족하면서, 핵심 플로우(작품 선택 → 필사 → 결과)를 완성하는 것이다.

## 1) 용어

- 공개 작품(Public Work): 관리자가 업로드한 “저작권 만료 작품” 텍스트
- 내 작품(My Work): 사용자가 직접 업로드한 텍스트(단, MVP에서는 **로컬 저장만**)
- 결과(Result): 1회 필사 세션의 요약 지표(정확도/WPM/시간 등)

## 2) 공통 제약 / 비기능 요구사항

### 2.1 무조건 무료(핵심)

- 결과/내 작품은 **서버 저장 금지**(로컬 저장: IndexedDB 우선, LocalStorage는 보조)
- 서버 비용을 증가시키는 기능(동기화/랭킹/공유/대용량 업로드)은 MVP에서 제외
- 운영 인프라는 “무료 플랜만 사용”을 원칙으로 하며, **자동 과금이 발생하지 않아야 한다**
  - 무료 한도 초과 시 과금이 아니라 기능 제한/차단으로 동작해야 한다(하드 캡)
- MVP(개발/배포)는 **정적 사이트(서버 런타임/DB 없음)** 를 원칙으로 한다.
- MVP(개발/배포)는 **AWS(Amplify/Cognito/AppSync/S3) 등 종량 과금형 인프라를 사용하지 않는다**.

### 2.2 성능/UX

- 타이핑 입력은 지연이 없어야 함(키 입력 → UI 반영이 즉시)
- 오프라인/저신뢰 네트워크에서도 “내 작품/내 기록 조회”는 동작해야 함(로컬 기반)

### 2.3 보안/프라이버시

- 사용자 입력 텍스트(필사 내용)는 서버로 전송하지 않는다(**GA4 포함**)
- 공개 작품은 저작권 만료 작품만 취급하며 출처/근거 메타데이터를 남긴다

### 2.4 SEO(랜딩 유입)

> “랜딩 유입”이 핵심이므로, SEO는 랜딩(마케팅) 페이지 중심으로 챙긴다.  
> 다만 “무조건 무료(0원 보장)” 제약 때문에 MVP에서는 SSR(요청당 서버 렌더링)을 사용하지 않는다.

- 랜딩 페이지는 정적 HTML로 제공되어야 한다(SSG 또는 정적 export)
- 기본 SEO 메타데이터가 있어야 한다
  - `title`, `meta description`, canonical
  - Open Graph(`og:*`) 및 공유 이미지(OG image)
- `robots.txt` 및 `sitemap.xml`을 제공해야 한다
- 검색(SEO) 대상은 **랜딩 페이지만** 포함한다(필사/기록/라이브러리 등 앱 기능 화면은 비대상).
- 앱 기능 화면은 검색 엔진에 인덱싱되지 않아야 한다(예: `noindex` 메타 태그 또는 `robots.txt` 차단).

**수용기준**
- 자바스크립트가 느리게 로드되거나 일부 차단되어도, 랜딩 페이지의 핵심 카피/CTA가 보인다(정적 HTML)
- 공유(카카오/슬랙 등) 시 OG 정보가 정상 표시된다
- `/robots.txt`, `/sitemap.xml`이 200으로 응답한다
- 랜딩 이외의 앱 기능 화면은 검색 엔진에 인덱싱되지 않도록 설정되어 있다(예: `noindex`).

## 3) 사용자/권한 요구사항

### R-AUTH-01 게스트(비로그인)

- 게스트는 공개 작품을 선택해 필사할 수 있다.
- 게스트는 “내 작품”을 업로드/관리할 수 있다(로컬 저장).
- 게스트의 결과/기록은 로컬에 저장된다.

**수용기준**
- 로그인하지 않아도 필사 화면 진입 가능
- 필사 완료 후 결과가 “이 기기”에 저장되고, 브라우저 새로고침 후에도 조회 가능
- 로그인하지 않아도 내 작품 업로드가 가능하고, 새로고침 후에도 목록이 유지된다(IndexedDB)

### R-AUTH-02 로그인/회원가입 — MVP 제외

MVP는 “0원 보장 + 마찰 최소”를 위해 로그인/회원가입을 제공하지 않는다.
- 내 작품/결과는 로컬 저장이므로, 계정 없이도 핵심 기능이 성립한다(ADR 0003).
- (후속) 기기간 동기화/서버 저장이 필요해질 때에만 로그인 도입을 검토한다(별도 ADR).

**수용기준**
- 어떤 화면도 로그인/회원가입을 요구하지 않는다(핵심 플로우에 인증 게이트 없음)

### R-AUTH-03 관리자(콘텐츠)

MVP는 정적 사이트이므로, 공개 작품 업로드/관리는 앱 내부 “관리자 페이지”가 아니라 운영 프로세스로 처리한다.
- 관리자는 공개 작품(정적 파일/인덱스)을 추가/삭제/갱신할 수 있어야 한다.

**수용기준**
- 작품 파일/인덱스를 갱신해 배포하면, 라이브러리 목록에 반영된다
- 작품 메타데이터에 저작권 만료 근거/출처가 포함된다(운영 체크)

## 4) 공개 작품 라이브러리 요구사항

### R-LIB-01 작품 목록 조회

- 사용자는 공개 작품 목록을 조회할 수 있다(제목/저자 최소 정보).

**수용기준**
- 최초 진입 시 작품 목록이 로드된다
- 목록은 제목/저자로 구분되어 표시된다

### R-LIB-02 작품 검색/필터(최소)

- 사용자는 제목/저자로 검색할 수 있다(최소 기능).

**수용기준**
- 검색어 입력 후 목록이 필터링된다
- 검색 결과가 없으면 “결과 없음” 상태가 표시된다

### R-LIB-03 작품 선택

- 사용자는 작품을 선택하고 필사 화면으로 이동할 수 있다.

**수용기준**
- 작품 상세(또는 시작 버튼)에서 필사 시작이 가능
- 선택한 작품의 텍스트를 필사 화면에서 로딩한다

### R-LIB-04 공개 작품 소스(works origin)

- 공개 작품 인덱스/원문은 “작품 전용 정적 호스팅(works origin)”에서 로드한다(앱 배포와 독립적으로 업데이트).
- 앱은 `worksBaseUrl`을 기준으로 아래 리소스를 fetch 한다.
  - 인덱스: `{worksBaseUrl}/works/index.json`
  - 원문: `{worksBaseUrl}/works/{workId}.txt` (또는 인덱스에 정의된 part 경로)
- 대용량 작품은 단일 파일이 아니라 “파트(장/권/분할 파일)”로 제공할 수 있어야 한다.
  - 앱은 필요한 파트만 다운로드한다(최소: 첫 파트).

**수용기준**
- `worksBaseUrl`이 설정된 경우, 해당 origin에서 작품 목록/원문을 정상 로드한다(브라우저 콘솔에 CORS 오류가 없어야 함)
- 파트가 있는 작품은 “첫 파트만” 다운로드 후 필사 시작이 가능하다

## 5) 필사(타이핑) 요구사항

> 오타는 “진행 허용 + 즉시 강조(빨간색) + 페이지 종료 리포트”로 합의되었다.  
> 페이지(Chunk) 단위는 **문단**으로 정의한다.  
> 판정/편집 규칙(공백/문장부호/줄바꿈/Backspace 정책)은 ADR 0001에 따라 본 문서에 고정한다.

### R-TYPE-01 텍스트 표시

- 필사 대상 텍스트를 **문단(=페이지/Chunk)** 단위로 표시한다.
- 타이핑 UI는 “고스트 텍스트(원문)” 위에 입력이 덮어씌워지는 느낌(overlay)으로 렌더링한다.

**수용기준**
- 사용자가 “지금 입력해야 하는 구간”을 명확히 인지할 수 있다
- 미입력 구간은 회색(고스트)으로 보이고, 입력한 구간은 더 진한 색으로 보인다
- 오타가 난 글자는 빨간색으로 표시된다(경고/모달 없이)

### R-TYPE-02 입력 비교/피드백

- 사용자의 입력을 목표 텍스트와 비교해 시각적 피드백을 제공한다.
- 오타가 발생해도 **입력을 계속 허용**한다.
- 오타가 발생한 위치는 즉시 “오타 상태”로 표시한다(예: 빨간색).
- 판정/편집 정책(기본)
  - 공백(space)과 줄바꿈(newline)은 목표 텍스트와 **완전 일치**해야 한다.
  - 구두점/대문자도 판정한다(옵션 기본값: **ON**).
  - Backspace로 수정 가능하며, 수정되어 더 이상 불일치가 남지 않으면 오타로 집계하지 않는다.
  - 커서는 임의 이동 없이 “현재 진행 위치”에서만 입력한다.
  - 필사 화면에서는 한글 IME 조합 입력과 일반 키보드 입력만 허용한다.
  - 필사 화면에서는 붙여넣기(`Ctrl/Cmd+V`), 드래그앤드롭 텍스트 투입, 스크립트/매크로성 다문자 삽입을 허용하지 않는다.

**수용기준**
- 올바른 입력/오타가 즉시 구분되어 표시된다
- 오타가 발생해도 커서/진행이 멈추지 않는다
- 공백/줄바꿈/구두점/대문자 판정이 옵션/정책에 따라 일관되게 동작한다
- Backspace로 수정하면 해당 오타 표시가 즉시 해제되고, 문단 리포트에서도 제외된다
- 사용자가 임의로 커서를 옮겨 중간을 편집할 수 없다(현재 위치 입력만)
- 한글 IME 조합 입력은 정상적으로 처리된다
- 필사 화면에서 붙여넣기/드래그앤드롭/비정상 다문자 입력은 반영되지 않는다

### R-TYPE-03 시간 측정 및 지표 계산

- 필사 시작/종료를 기준으로 경과 시간을 측정한다.
- 결과로 최소 아래 지표를 계산한다.
  - 정확도(accuracy)
  - WPM
  - 소요시간(elapsedTime)

**수용기준**
- 완료 시 결과 요약 화면에 위 지표가 표시된다
- 계산 로직은 동일 입력에 대해 일관된 결과를 낸다

### R-TYPE-04 일시정지/재시작

- 사용자는 필사를 일시정지/재개할 수 있다.

**수용기준**
- 일시정지 중에는 타이머가 증가하지 않는다
- 재개하면 동일 지점에서 계속할 수 있다

### R-TYPE-05 페이지 종료 오타 리포트

- “문단(=페이지/Chunk) 단위”가 끝나면, 해당 문단에서 발생한 오타를 요약해 보여준다.

**수용기준**
- 문단(=페이지) 완료 시 오타 개수/요약이 표시된다
- “어떤 구간에서 오타가 났는지” 사용자가 재확인할 수 있다(예: 하이라이트/목록)

### R-TYPE-06 진행 상태 자동 저장/이어하기

- 필사 중인 세션은 로컬에 자동 저장되어야 한다.
- 새로고침/탭 종료/브라우저 재방문 이후에도, 같은 기기에서는 저장된 세션을 이어서 진행할 수 있어야 한다.
- 이어하기는 “현재 작품/파트/문단 위치 + 현재 문단 입력값 + 누적 시간 + 설정 스냅샷”을 기준으로 복원한다.
- 완료된 세션의 드래프트는 제거한다.
- 사용자는 저장된 드래프트를 버리고 처음부터 다시 시작할 수 있어야 한다.

**수용기준**
- 필사 도중 새로고침 후 진입하면 “이어하기” UI가 노출된다
- 이어하기를 선택하면 마지막 저장 지점의 문단/입력 상태/누적 시간으로 복원된다
- 새로 시작하기를 선택하면 기존 드래프트가 삭제되고 처음 문단부터 시작한다
- 작품 완료 후에는 해당 드래프트가 남지 않는다

### R-OPT-01 가독성 옵션(폰트/테마) — MVP 포함

- 사용자는 타이핑 화면에서 가독성을 위해 폰트 크기/테마(다크 모드 등)를 조정할 수 있다.

**수용기준**
- 폰트 크기를 변경해도 입력/지표 계산이 깨지지 않는다
- 다크 모드에서 대비가 충분해 읽기/입력이 가능하다

### R-OPT-02 판정 옵션(구두점/대문자) — MVP 포함

- 사용자는 구두점/대문자 판정을 켜거나 끌 수 있다(특히 외국문학).
- 옵션 기본값은 **ON** 이다.

**수용기준**
- 옵션 on/off에 따라 정확도/오타 계산 기준이 일관되게 바뀐다
- 최초 진입(기본 설정)에서는 ON 상태로 시작한다

### R-OPT-03 오타 표시 방식 — MVP 포함

- 사용자는 오타 표시 스타일(예: 색상/밑줄 등)을 선택할 수 있다.

**수용기준**
- 표시 방식 변경이 성능(입력 지연)에 영향을 주지 않는다

## 6) 결과(로컬 저장) 요구사항

### R-RES-01 결과 저장

- 필사 완료 시 결과를 로컬에 저장한다.

**수용기준**
- 저장된 결과는 새로고침/재접속 후에도 유지된다(IndexedDB 우선)
- 저장 실패 시 사용자에게 오류가 표시되고, 재시도 가능하다

### R-RES-02 결과 조회(히스토리)

- 사용자는 과거 결과 목록을 조회할 수 있다.

**수용기준**
- 날짜/작품 기준으로 최소한의 정렬/필터가 가능하다(최소 1개 방식)

### R-RES-03 진행 중 세션 드래프트 저장

- 결과와 별도로, 진행 중인 세션 드래프트를 IndexedDB에 저장한다.
- 드래프트는 “완료된 결과”와 구분되어 관리되어야 한다.

**수용기준**
- 진행 중 세션 드래프트가 결과 히스토리와 섞여 표시되지 않는다
- 같은 작품에 저장된 드래프트가 있으면, 앱이 이를 감지해 이어하기 진입점을 제공한다

## 7) 내 작품(로컬 업로드) 요구사항

### R-MY-01 내 작품 업로드

- 사용자는 텍스트를 업로드하여 “내 작품”으로 저장할 수 있다.
- 업로드 방식은 `.txt` 파일 선택 또는 텍스트 붙여넣기를 지원한다.
- 이 허용 범위는 “내 작품 업로드 화면”에만 적용되며, 필사 화면의 붙여넣기 금지 정책과 분리한다.

**수용기준**
- 업로드한 텍스트는 IndexedDB에 저장된다
- 업로드한 작품을 선택해 동일한 필사 경험을 사용할 수 있다
- `.txt` 파일 선택과 텍스트 붙여넣기 중 어느 방식으로 추가해도 같은 데이터 구조로 저장된다

### R-MY-02 비용 방지

- 내 작품은 MVP에서 서버/Storage에 업로드하지 않는다.

**수용기준**
- 네트워크 요청 없이도 내 작품이 동작한다(로컬만)

## 8) 분석(DAU) 요구사항

### R-ANA-01 DAU 측정

- GA4를 통해 DAU를 확인할 수 있어야 한다.
- DAU는 “필사 시작” 이벤트(`typing_start`)를 발생시킨 유니크 사용자 수(일별)로 정의한다(ADR 0011).

**수용기준**
- GA4에서 “일별 `typing_start` 이벤트 유저 수”를 확인할 수 있다(랜딩만 보고 이탈한 방문 제외)
- 이벤트/파라미터에 사용자 필사 텍스트(콘텐츠) 또는 토큰/시크릿이 포함되지 않는다

### R-ANA-02 이벤트 스키마(최소)

> 이벤트명/파라미터는 ADR 0011을 기준으로 구현한다.

- 이벤트(최소)
  - `typing_start` (DAU)
  - `typing_paragraph_complete`
  - `typing_complete`
  - `my_work_upload`
- 공통 파라미터(권장)
  - `work_kind`: `public` | `my`
  - `public_work_id`: 공개 작품 id(`work_kind=public`일 때만)
  - `work_language` (선택)
  - `punctuation_case_on` (선택)
  - `app_version` (선택)

**수용기준**
- `typing_start`는 한 세션에서 1회만 전송된다(중복 방지, 이어하기 복원 시 재전송하지 않음)
- `my_work_upload`는 내 작품 저장 성공 시 1회 전송된다(파일명/원문 텍스트 미전송)

## 9) 운영 가드레일(확정)

- 앱 배포물은 정적 산출물만 포함한다. SSR/Functions/DB/Storage는 MVP에서 금지한다.
- 호스팅은 `Firebase Hosting (Spark, Hosting만)` 을 기준으로 하되, 결제수단 연결 또는 유료 플랜 전환 요구가 확인되면 공개 런칭 전에 즉시 백업 호스트로 전환한다.
- 앱과 공개 작품은 분리 배포한다. 앱은 `worksBaseUrl`로만 공개 작품을 조회하고, 앱 빌드 산출물에 공개 작품 원문을 포함하지 않는다.
- 공개 작품 캐시 정책은 인덱스(`index.json`)와 원문(`*.txt` 또는 part 파일)을 구분한다.
  - 인덱스: 짧은 캐시 또는 `no-cache`
  - 원문/파트 파일: 버전 또는 해시가 반영된 파일명 + 장기 캐시
- 사용자 데이터(내 작품/결과/드래프트)는 전부 로컬 저장으로 유지한다.
- 초기 운영 도메인은 무료 서브도메인만 사용한다.

**수용기준**
- 앱 배포 후 네트워크 탭 기준, 공개 작품 원문은 앱 origin이 아니라 `worksBaseUrl`에서 로드된다
- 공개 작품 인덱스 갱신 시 앱 재배포 없이 목록 반영이 가능하다
- 공개 런칭 체크리스트에 “결제수단/유료 플랜 요구 없음” 확인 항목이 포함된다
